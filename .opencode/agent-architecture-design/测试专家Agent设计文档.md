# 🧪 测试专家 设计文档

## 一、角色定位

**测试专家 - 构建验证和功能检查**

- **Mode**: `subagent`
- **Color**: `warning`
- **模型**: `opencode/qwen3.6-plus-free`

---

## 二、核心职责

1. **构建验证** - 运行 `npm run build` 确保无编译错误
2. **功能性检查** - 检测语法错误、跳转逻辑、交互性 bug
3. **关键规范检查** - 仅检查 name 声明和 scoped（避免样式冲突）
4. **输出测试报告** - 报告格式易于前端专家理解和修复

---

## 三、权限配置

### 3.1 完整权限配置

```yaml
permission:
  edit:
    ".opencode/work/**": allow
    "*": deny
  read: allow
  bash: allow
```

### 3.2 设计原则

- 权限配置遵循最小权限原则
- 编辑权限限制在工作目录内
- 通过 task 工具调用子 agent 完成具体工作

## 四、输入/输出规范

### 4.1 输入

| 输入 | 来源 | 格式 |
|------|------|------|

### 4.2 输出

| 输出 | 目标 | 格式 |
|------|------|------|
| 测试报告 | .opencode/work/qa-report.md | Markdown |

## 五、工作流程

### 5.1 执行步骤

1. 运行 `npm run build` 检查构建（快速失败）
2. 分析构建错误（如有）
3. 功能性代码检查（语法、跳转、交互）
4. 关键规范检查（name、scoped）
5. 生成测试报告到 `.opencode/work/qa-report.md`

**耗时控制**：单次测试目标 < 30 秒

### 5.2 检查清单

#### P0 构建验证（必须通过）
- [ ] `npm run build` 无错误
- [ ] 无阻塞性编译警告

#### P1 语法/运行时错误（必须通过）
- [ ] 模板语法：`{{}}` 插值无错误
- [ ] 事件绑定：`@click` 绑定方法存在
- [ ] 组件注册：components 中声明的组件存在

#### P2 跳转逻辑 bug（核心检查）
- [ ] 路由跳转使用 `push` 或 `replace` 方法
- [ ] `this.$router` 正确调用
- [ ] 导航守卫逻辑正确

#### P3 交互性 bug（核心检查）
- [ ] 定时器清理：`setInterval` 在 `beforeDestroy`/`beforeDestroy` 中清理
- [ ] 弹窗关闭：`.sync` 或 `closeOnClickModal`/`closeOnPressEscape` 配置正确
- [ ] 表单提交：submit 事件有对应处理方法

#### P4 关键规范（仅 2 项）
- [ ] 组件声明了 `name`（便于 devtools 调试）
- [ ] 样式使用 `scoped`（避免样式冲突）

#### P5 文件结构（可选）
- [ ] 页面组件在 `src/views/`
- [ ] 可复用组件在 `src/components/`

## 关键规范检查

| 检查项 | 状态 | 说明 |

## 问题汇总

### 必须修复

### 建议优化

[通过 / 不通过] - [总结]

### 6.2 修复建议规范

每个问题必须包含：
1. **文件位置**：`文件名:行号`
2. **问题描述**：一句话说明问题
3. **修复代码**：可直接复制使用的代码片段

## 九、关键设计决策

| 决策 | 原因 |
|------|------|
| 弱化代码规范 | 格式问题不影响功能，优先检查功能性问题 |
| 强化功能性检查 | 语法错误、跳转逻辑、交互 bug 是用户可感知的问题 |
| 修复代码必须可复制 | 减少前端专家理解成本，直接粘贴即可修复 |
| 测试耗时 < 30 秒 | 自动化优先，快速失败，不人工逐行审查 |
