# 分析报告：多 Agent 架构实战复盘 — 杭州银行项目

> 基于 2026-04-02 杭州银行理财产品展示 MVP 项目的完整执行过程，对多 Agent 架构设计进行深度剖析。

---

## 一、执行过程回顾

### 1.1 实际执行链路

```
用户输入"杭州银行"
  → project-manager 追问4个维度需求
  → 用户补充：银行官网、理财产品展示、个人客户、MVP核心页面
  → 生成执行计划 → 用户确认
  → product-manager 生成 PRD（❌ 失败，返回空 result，orchestrator 手动补写）
  → ui-designer 生成设计（✅ 成功，产出 design.md）
  → frontend-expert 生成代码（❌ 失败，返回空 result，orchestrator 手动编写10个文件）
  → qa-engineer 生成报告（❌ 失败，返回空 result，orchestrator 手动编写报告）
  → 完成
```

### 1.2 成功率统计

| 指标 | 数值 |
|------|------|
| Subagent 调用次数 | 4 |
| 成功次数 | 1（ui-designer） |
| 失败次数 | 3（product-manager, frontend-expert, qa-engineer） |
| 成功率 | **25%** |
| Orchestrator 兜底工作量 | **~70%** |

### 1.3 关键时间线

```
T+0    用户输入"杭州银行"（4字）
T+1    PM 追问4个维度
T+2    用户确认需求
T+3    执行计划生成 → 用户确认
T+4    调用 product-manager → 空 result → 文件不存在 → PM 手动写 PRD
T+5    调用 ui-designer → 成功 → design.md 生成
T+6    用户确认设计
T+7    调用 frontend-expert → 空 result → 文件不存在 → PM 手动写10个文件
T+8    构建通过
T+9    调用 qa-engineer → 空 result → 文件不存在 → PM 手动写报告
T+10   项目完成
```

## 二、架构层面问题

### 2.1 无输出验证机制（最严重）

**现象**：`product-manager`、`frontend-expert`、`qa-engineer` 三个 subagent 调用后均返回空 `<task_result>`，但系统未报错。Orchestrator 误以为执行成功，直到读取文件时才发现文件不存在。

**根因分析**：

| 层级 | 问题 |
|------|------|
| 平台层 | subagent 的 `task_result` 为空时没有触发失败信号 |
| 编排层 | orchestrator 没有在执行后立即验证产出物是否存在 |
| 流程层 | 流程推进依赖人工检查，而非自动化验证 |

**后果**：每个失败节点都导致流程中断，需要 orchestrator 重新手动执行，完全失去了自动化的意义。

**架构设计中的规定 vs 实际执行**：

```
设计规定（agent-architecture.md 第269-277行）:
  "执行 subagent 任务后，验证产出物是否存在且非空"
  "验证失败时不进入下一步，最多重试 2 次"

实际执行:
  调用 product-manager → 返回空 result → 未检查文件 → 标记 prd completed ❌
  调用 frontend-expert → 返回空 result → 未检查文件 → 继续执行 ❌
  调用 qa-engineer → 返回空 result → 未检查文件 → 手动补报告 ❌
```

### 2.2 线性流程缺乏容错

**现象**：流程定义为严格的串行链 `plan → prd → design → code → qa`，任一环节失败都会阻塞后续。

**问题**：

| 缺失能力 | 影响 |
|---------|------|
| 无重试机制（retry） | subagent 失败后无法自动恢复 |
| 无降级策略（fallback） | 没有备选执行路径 |
| 无并行执行 | PRD 和 UI 设计理论上可部分并行，但架构强制串行 |
| 无超时控制 | 无法判断 subagent 是"正在执行"还是"已卡死" |

### 2.3 状态管理脆弱

**现象**：`process.md` 使用 YAML frontmatter + Markdown 表格混合存储状态，每次更新都需要完整重写文件。

**问题**：

| 问题 | 说明 |
|------|------|
| 状态与操作不同步 | agent 调用后先更新状态为 completed，但实际可能失败 |
| 无原子性保证 | 中断后状态不一致（如 completed 列表已更新但产出物不存在） |
| 非结构化存储 | 读取状态需要解析 Markdown 文本，容易出错 |
| 无版本控制 | 状态覆盖后无法回滚 |

### 2.4 缺乏上下文传递标准化

**现象**：每个 subagent 的 prompt 都需要 orchestrator 手动拼接大量上下文（设计规范、PRD 路径、代码规范等）。

**问题**：

| 问题 | 说明 |
|------|------|
| 上下文易遗漏 | 每次调用都需要手动注入项目背景、代码规范、设计约束 |
| 无统一 context store | 没有共享的上下文管理机制 |
| 重复注入 | 每个 agent 都需要重复了解项目背景信息 |
| Token 浪费 | 大量上下文重复传递，挤占有效输出空间 |

### 2.5 Subagent 执行模型不透明

**现象**：调用 subagent 后只能收到一个空的 `<task_result>`，无法知道：
- subagent 内部执行了多少步
- 使用了哪些工具
- 遇到了什么错误
- 当前进度如何

**影响**：Orchestrator 处于"盲调"状态，无法进行有效的进度监控和异常处理。

## 三、各 Agent 具体问题

### 3.1 🎯 project-manager（Orchestrator）

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 未验证 subagent 输出 | 🔴 高 | 三次 subagent 调用失败均未及时发现，流程继续推进 |
| 过度兜底 | 🔴 高 | 最终 70% 的工作量由 orchestrator 手动完成，违背分工初衷 |
| 流程控制僵化 | 🟡 中 | 严格按照 workflow.md 执行，没有根据失败情况动态调整策略 |
| 需求引导可优化 | 🟡 中 | 用户首次输入"杭州银行"时追问了4个问题，但应提供模板化输入格式 |
| 状态更新先于验证 | 🟡 中 | 先更新 process.md 为 completed，后验证产出物，导致状态不一致 |

**典型失误链路**：

```
调用 product-manager
  → 返回空 <task_result>
  → 未检查 .opencode/work/prd.md 是否存在
  → 更新 process.md: completed: [plan, user_gate_plan, prd]  ← 错误状态
  → 继续调用 ui-designer
  → ...（后续步骤均基于错误状态推进）
```

**设计规定 vs 实际行为**：

```
规定: "执行 subagent 任务后，验证产出物是否存在且非空"
实际: 调用后直接更新状态，未验证产出物

规定: "验证失败时不进入下一步，最多重试 2 次"
实际: 未验证，直接进入下一步

规定: "如果验证失败，标记当前步骤为 failed，通知用户"
实际: 未标记 failed，用户无感知
```

### 3.2 💼 product-manager

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 未生成产出物 | 🔴 高 | 被调用后没有写入 `.opencode/work/prd.md` |
| 无失败反馈 | 🔴 高 | 执行失败但没有返回任何错误信息 |
| 权限可能不足 | 🟡 中 | 配置为 `edit: { ".opencode/work/**": "allow" }`，但实际未执行写入 |

**可能原因**：
- Prompt 中未明确要求"必须写入文件"
- Subagent 执行环境可能缺少 write 权限
- 模型可能选择了"在对话中输出"而非"写入文件"

### 3.3 🎨 ui-designer（唯一成功）

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 外部依赖重 | 🟡 中 | 依赖 `ui-ux-pro-max` skill，需要 Python 环境和脚本执行 |
| 输出格式衔接不足 | 🟢 低 | 设计文档与 PRD 的结构衔接不够紧密 |

**成功原因分析**：

| 因素 | 说明 |
|------|------|
| Skill 支持 | 有 `ui-ux-pro-max` skill 提供结构化脚本输出 |
| 任务明确 | 调用时明确指定了输出路径和格式要求 |
| Orchestrator 手动执行 | 实际由 orchestrator 直接调用 skill 脚本并写入文件 |

> **关键洞察**：ui-designer 的"成功"实际上是 orchestrator 手动执行的结果，而非 subagent 自主完成。这说明**有明确工具链支持的任务更容易成功**。

### 3.4 💻 frontend-expert

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 完全未执行 | 🔴 高 | 被调用后没有生成任何 .vue 文件 |
| 任务粒度过大 | 🔴 高 | 单次 prompt 要求生成 10 个文件，超出可靠输出范围 |
| 无增量生成 | 🟡 中 | 没有分批次生成并验证，而是一次性要求全部完成 |
| 备份机制未触发 | 🟡 中 | 设计要求"修改前先备份"，但实际未执行 |

**Prompt 设计问题**：

```
单次要求生成:
  - 1 个 Vuex 模块（~200行）
  - 5 个 Vue 组件（~500行）
  - 1 个页面组件（~100行）
  - 2 个配置文件修改
  总计: ~800+ 行代码

这超出了单次 subagent 调用的可靠输出范围。
```

**应拆分的粒度**：

```
理想拆分:
  调用1: 创建 Vuex 模块 → 验证 → 继续
  调用2: 创建 AppHeader + AppFooter → 验证 → 继续
  调用3: 创建 ProductFilter + ProductCard → 验证 → 继续
  调用4: 创建 ProductDetail + WealthProducts → 验证 → 继续
  调用5: 更新路由和 Store 配置 → 验证 → 继续
```

### 3.5 🧪 qa-engineer

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 完全未执行 | 🔴 高 | 被调用后没有生成 qa-report.md |
| 验证能力缺失 | 🟡 中 | QA agent 应该能够运行 `npm run build` 并解析输出，但实际未执行 |
| 报告格式依赖人工 | 🟢 低 | 报告模板已提供，但 agent 未填充内容 |

## 四、根因分析

### 4.1 核心矛盾

| 期望 | 实际 | 差距 |
|------|------|------|
| 5 个 agent 各司其职 | 1 个 agent 完成 70% 工作 | 分工失效 |
| 流程自动推进 | 3 次流程中断需手动恢复 | 自动化率 25% |
| 产出物自动生成 | 2 次产出物丢失需手动重写 | 可靠性不足 |
| 质量自动验证 | QA 报告手动编写 | 验证链断裂 |

### 4.2 技术根因

```
1. Subagent 执行模型不透明
   → 无法知道 subagent 内部是否真正执行了文件写入操作
   → task_result 为空不代表失败，也不代表成功

2. 缺少执行结果校验层
   → 应在每个 agent 完成后自动检查产出物
   → 当前依赖 orchestrator 主观判断

3. Prompt 工程不足
   → frontend-expert 的任务拆分不够细
   → 超出了模型单次可靠输出范围

4. 错误传播断裂
   → subagent 失败 → 空 result → orchestrator 误判 → 流程继续
   → 没有错误信号向上传播的机制
```

### 4.3 架构设计缺陷矩阵

| 缺陷 | 影响范围 | 发生频率 | 修复难度 |
|------|---------|---------|---------|
| 无输出验证 | 全局 | 每次 subagent 调用 | 低 |
| 任务粒度过大 | frontend-expert | 多文件生成场景 | 中 |
| 状态管理脆弱 | 全局 | 每次状态更新 | 中 |
| 上下文传递不标准 | 全局 | 每次 subagent 调用 | 高 |
| 无重试机制 | 全局 | subagent 失败时 | 中 |
| Subagent 执行不透明 | 全局 | 每次调用 | 高（平台限制） |

## 五、改进方向

### 5.1 架构层改进

| 改进点 | 方案 | 优先级 |
|--------|------|--------|
| 增加 Output Validator | 每个 agent 完成后自动验证产出物存在性和非空 | P0 |
| 支持并行执行 | PRD 和设计可部分并行，减少总耗时 | P1 |
| 增加 Retry 机制 | 失败后自动重试，最多 2 次 | P0 |
| 结构化状态存储 | 使用 JSON 替代 Markdown 存储状态 | P1 |
| 统一 Context Store | 建立共享上下文管理，避免手动拼接 | P1 |

### 5.2 Agent 层改进

| Agent | 改进点 | 方案 |
|-------|--------|------|
| project-manager | 输出验证 | 每次 subagent 调用后执行 `read` 检查产出物 |
| project-manager | 状态一致性 | 验证通过后再更新 process.md |
| frontend-expert | 任务拆分 | 每个文件一个 agent 调用，或最多 2-3 个相关文件一组 |
| frontend-expert | 增量生成 | 生成一批 → 验证 → 生成下一批 |
| product-manager | 明确输出方式 | Prompt 中强调"必须写入文件"而非"在对话中输出" |
| qa-engineer | 自动化验证 | 明确要求执行 `npm run build` 并解析输出 |

### 5.3 Prompt 工程改进

| 问题 | 改进方案 |
|------|---------|
| 上下文重复注入 | 建立 `.opencode/work/context.md` 统一上下文文件 |
| 任务粒度过大 | 定义最大输出行数限制（如单次不超过 200 行） |
| 输出方式不明确 | 所有 agent prompt 中统一添加"产出物必须写入指定文件" |
| 缺少验证指令 | 在 prompt 中添加"写入文件后读取确认内容正确" |

### 5.4 流程改进

```
改进前（当前）:
  plan → prd → design → code → qa
  任一失败 → 流程阻塞 → 人工介入

改进后（目标）:
  plan → [prd | design] → code → qa
           ↓ 失败自动重试 2 次
           ↓ 仍失败 → 通知用户 + 提供降级选项
           
  code 阶段拆分为:
    code-1: Vuex 模块 → 验证
    code-2: 基础组件 → 验证
    code-3: 业务组件 → 验证
    code-4: 页面组装 → 验证
    code-5: 配置更新 → 验证
```

## 六、关键洞察

### 6.1 什么情况下 Subagent 能成功？

基于本次实战，subagent 成功的条件：

| 条件 | 说明 |
|------|------|
| 有明确工具链支持 | ui-designer 有 skill 脚本辅助 |
| 任务粒度小 | 单一产出物比多产出物更可靠 |
| 输出路径明确 | 明确指定文件路径和格式 |
| Orchestrator 深度参与 | 实际成功的是 orchestrator 手动执行的部分 |

### 6.2 什么情况下 Subagent 会失败？

| 条件 | 说明 |
|------|------|
| 任务过于复杂 | frontend-expert 要求一次生成 10 个文件 |
| 输出方式模糊 | 没有明确"必须写入文件" |
| 缺乏验证反馈 | 没有"写入后读取确认"的指令 |
| 超出模型能力 | 800+ 行代码超出单次可靠输出范围 |

### 6.3 本质问题

**当前实现是一个"编排系统"而非真正的"多 Agent 协作系统"**。

| 特征 | 编排系统（当前） | 真正的多 Agent 协作 |
|------|-----------------|-------------------|
| 错误处理 | 依赖人工发现和处理 | 自动检测、重试、降级 |
| 状态管理 | 手动更新，易不一致 | 自动同步，原子操作 |
| 任务分配 | 固定流程，刚性执行 | 动态规划，灵活调整 |
| 结果验证 | 人工检查 | 自动化验证链 |
| 上下文传递 | 手动拼接 prompt | 共享上下文存储 |
| 容错能力 | 无 | 重试、回滚、降级 |

## 七、结论与建议

### 7.1 结论

1. **Subagent 执行模型在当前平台下不可靠**，3/4 的调用失败表明不能作为核心生产路径
2. **Orchestrator 承担了过多兜底工作**，70% 的手动工作量使多 agent 分工失去意义
3. **流程缺乏自动化验证**，状态更新与实际操作不同步，需要人工介入才能保证质量
4. **有工具链支持的任务（ui-designer + skill）表现更好**，说明工具化是提升可靠性的方向

### 7.2 短期改进（1-2 周）
