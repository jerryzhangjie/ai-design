# 多 Agent 协作架构深度剖析（第二版）

在上一版分析的基础上，结合**实际执行过程中的对话记录、文件内容、代码实现**进行更深入、更尖锐的剖析。

---

## 一、流程全链路复盘：信息是如何一步步失真的

```
用户输入: "杭州银行官网"
  ↓ 失真 1: 我替用户做了大量假设
计划: "首页 + 核心模块 + 基础交互"
  ↓ 失真 2: PRD 没有验证假设的合理性
PRD: 7 个模块，全部 Must Have
  ↓ 失真 3: 设计规范与 PRD 组件映射模糊
设计: 9 个组件，40+ CSS Token
  ↓ 失真 4: 前端工程师选择性实现
代码: 7 个组件，部分交互简化
  ↓ 失真 5: QA 只检查了静态代码
QA: 构建通过 = 质量通过
```

**核心发现**：每一层都在"合理简化"，但简化累积起来，最终产物与用户原始意图的偏差已经很大。用户说的是"杭州银行官网"，最终交付的是一个**通用的银行首页模板**，而非杭州银行的官网。

---

## 二、架构层面的深层问题

### 1. Agent 角色定义存在根本性矛盾

当前架构定义：

| Agent | 定义 | 实际行为 |
|-------|------|---------|
| project-manager | "流程编排，用户第一接触点" | 变成了 Prompt 路由器 |
| product-manager | "需求分析，PRD 输出" | 变成了模板填充器 |
| ui-designer | "视觉设计，组件树，布局" | 变成了 CSS Token 生成器 |
| frontend-expert | "Vue 2 组件开发" | 变成了代码批量生成器 |
| qa-engineer | "构建验证，代码审查" | 变成了缩进检查器 |

**问题本质**：每个 Agent 的职责定义是**产出物导向**的，而非**能力导向**的。这导致：
- product-manager 不需要理解银行行业，只需要输出 PRD 格式
- ui-designer 不需要理解用户体验，只需要输出设计规范格式
- qa-engineer 不需要保证质量，只需要输出检查报告格式

**这叫做"指标替代"——把手段当成了目的。**

---

### 2. 工作流定义是"伪敏捷"

查看 `.opencode/worker/workflow.md`：

```
用户请求 → plan → 确认 → prd → 确认 → design → 确认 → code → qa → done
```

这个流程自称"每步完成后等待用户确认"，但实际执行中：

| 确认节点 | 用户看到的内容 | 用户能做的决策 |
|---------|--------------|--------------|
| 计划确认 | 一段文字描述 | 只能"确认"或"提意见"，无法预览 |
| PRD 确认 | 摘要表格 | 303 行文档，用户不可能逐行审查 |
| 设计确认 | 摘要表格 | 没有线框图，用户无法想象最终效果 |

**问题本质**：确认节点设计为"审批式"而非"共创式"。用户在信息不充分的情况下被迫做出"确认"决策，这比不确认更危险——因为给了虚假的安全感。

---

### 3. 文件作为通信介质的致命缺陷

当前架构使用 `.opencode/work/` 下的 Markdown 文件作为 Agent 间通信介质，存在三个致命问题：

**问题 A：文件是静态的，需求是动态的**
- PRD 写完后，如果用户说"把登录改成扫码登录"，需要修改 PRD → 修改设计 → 修改代码
- 但实际执行中，没有任何机制保证这三个文件同步更新

**问题 B：文件是单向的，理解是双向的**
- ui-designer 读取 PRD，但 PRD 中不会记录"为什么选择这个模块布局"
- frontend-expert 读取设计规范，但规范中不会记录"为什么选择这个圆角值"
- 决策上下文丢失，后续修改只能猜测原始意图

**问题 C：文件是完整的，注意力是稀缺的**
- 给前端工程师的 Prompt 说"请先读取 PRD 文件（303 行）"
- 实际执行中，LLM 的上下文窗口有限，不可能完整吸收 303 行 + 设计文档 + 代码规范
- 结果就是**选择性实现**——只实现 Prompt 中明确提到的部分

---

### 4. 质量保障体系完全缺失

当前架构中，质量保障只有一个环节：最后的 `qa-engineer`。

但真正有效的质量保障应该是这样的：

```
需求质量 ← product-manager 自检 + 用户确认
  ↓
设计质量 ← ui-designer 对照 PRD 检查 + 用户预览确认
  ↓
代码质量 ← frontend-expert 自测 + 设计规范对照
  ↓
集成质量 ← qa-engineer 功能测试 + 回归测试
```

当前架构的问题：
- **没有质量门禁**：PRD 写得再烂也能流到设计阶段
- **没有回溯机制**：QA 发现问题后，只能记录在报告中，无法触发上游修改
- **没有自动化**：所有检查都是人工（Agent）代码审查，没有单元测试、E2E 测试

## 三、各 Agent 的深层问题（结合执行实际）

### 🎯 project-manager（我）

**执行中的实际表现**：
1. 用户说"杭州银行官网"，我直接给出了 4 个澄清问题 + "我的建议"
2. 用户说"按你的建议来"，我立刻制定了计划
3. 整个过程中，我没有做过一次**需求验证**

**深层问题**：

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| **确认偏误** | 用户说"杭州银行官网"，我立刻假设是"首页原型"，没有确认是否包含子页、后台、移动端 | 可能完全偏离用户真实需求 |
| **决策越权** | "我的建议"中直接决定了页面范围、核心模块、交互深度 | 用户变成了"橡皮图章" |
| **风险盲区** | 计划中没有识别任何风险（如：杭州银行官网的真实结构可能与假设不同） | 后续返工成本高 |
| **Prompt 工程缺失** | 给下游 Agent 的 Prompt 是自然语言描述，没有结构化约束、验收标准、边界条件 | 输出质量完全依赖 Agent 自身能力 |
| **缺少决策记录** | 没有记录"为什么选择首页而非多页"、"为什么选择蓝色系" | 后续无法追溯决策依据 |

**一句话总结**：我不是在"管理项目"，我是在"推进流程"。真正的 PM 应该做需求验证、风险管理、决策记录、利益相关者沟通，而我只是在问"下一步做什么"。

---

### 📋 product-manager

**执行中的实际表现**：
1. 输出了 303 行 PRD，格式完美
2. 但内容上，"用户场景"只有 3 条通用用户故事
3. 竞品分析完全缺失
4. 所有功能都是 Must Have

**深层问题**：

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| **模板驱动而非需求驱动** | 输出的是标准 PRD 模板的填充，而非针对"杭州银行"的深度分析 | 缺乏行业洞察 |
| **缺少真实数据支撑** | 理财产品的收益率区间、起购金额范围，都是随意定义的 | 原型与真实场景脱节 |
| **优先级失真** | 没有区分 MVP 和增强功能，导致前端工程师一次性实现所有功能 | 交付风险高 |
| **验收标准不可执行** | PRD 中的验收标准是"轮播图支持自动播放"，但没有定义"自动播放间隔多少"、"切换动画多长" | QA 无法验证 |
| **假设未验证** | 假设用户需要"个人/企业/金融市场"三个 Tab，但没有验证杭州银行官网实际是否有这个结构 | 可能完全错误 |

**一句话总结**：PRD 的"格式正确率"是 100%，但"需求准确率"可能不到 50%。product-manager 应该做需求验证和假设检验，而不是模板填充。

---

### 🎨 ui-designer

**执行中的实际表现**：
1. 输出了完整的设计规范，包含 40+ CSS Token
2. 但组件描述是通用的，没有针对杭州银行的具体场景
3. 没有输出任何可视化内容（线框图、布局示意图）

**深层问题**：

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| **设计系统 ≠ 页面设计** | 输出的是通用设计系统文档，而非"杭州银行首页"的具体设计 | 前端工程师需要自己推断布局 |
| **Token 与实现脱节** | 定义了 40+ 个 CSS 变量，但没有验证哪些是真正需要的 | 代码中可能只用了 20 个 |
| **缺少可视化沟通** | 纯文本描述"12 列栅格"，前端工程师很难想象最终效果 | 实现偏差大 |
| **响应式设计一笔带过** | 只提了"响应式行为"，没有定义具体断点和布局变化 | 移动端体验无法保证 |
| **动效规范不完整** | 只定义了时长，没有定义触发条件、连续行为、性能约束 | 交互体验不一致 |

**一句话总结**：ui-designer 输出的是"设计系统参考手册"，而非"这个页面的设计说明书"。设计师应该做视觉决策和沟通，而不是输出通用规范。

---

### 💻 frontend-expert

**执行中的实际表现**：
1. 一次性生成 7 个组件 + 1 个页面
2. 构建通过，功能覆盖率 100%
3. 但 Mock 数据硬编码、轮播图实现简陋、缺少错误处理

**深层问题**：

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| **批量生成而非迭代开发** | 一次性生成所有代码，没有分步验证 | 出错后调试成本极高 |
| **工程化思维缺失** | Mock 数据硬编码在页面中，没有分离为独立模块 | 后续无法替换为真实 API |
| **边界条件未处理** | 轮播图没有处理快速点击、登录表单没有处理网络错误 | 用户体验脆弱 |
| **样式管理混乱** | CSS 变量定义在页面组件根元素，子路由无法共享 | 扩展性差 |
| **真实场景考虑不足** | 登录没有"忘记密码"、没有验证码、没有状态管理 | 原型无法演进为真实产品 |

**一句话总结**：frontend-expert 追求的是"代码能跑"，而非"代码可维护"。工程师应该做工程决策，而不是代码生成。

---

### 🔍 qa-engineer

**执行中的实际表现**：
1. 运行了 `npm run build`，通过
2. 检查了代码规范（命名、缩进、scoped 样式）
3. 给出了 8.5/10 的评分

**深层问题**：

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| **构建通过 ≠ 质量通过** | 只检查了语法正确，没有检查功能正确 | 轮播图可能不自动播放，但构建依然通过 |
| **静态检查 ≠ 动态测试** | 没有运行页面验证交互行为 | Tab 切换可能不工作，但代码规范检查通过 |
| **评分依据模糊** | 8.5/10 没有给出计算公式 | 评分没有参考价值 |
| **问题修复无优先级** | 列出 9 个问题，没有说先修哪 3 个 | 开发团队不知道从哪里开始 |
| **没有自动化测试** | 只做了人工代码审查，没有编写测试用例 | 后续修改无法回归验证 |

**一句话总结**：qa-engineer 变成了"代码风格检查器"，而非"质量保障者"。QA 应该做功能验证和风险评估，而不是缩进检查。

## 四、根本性矛盾：LLM Agent 的能力边界

以上所有问题，归根结底是一个更深层的矛盾：

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   LLM 擅长：格式生成、模板填充、代码补全              │
│   LLM 不擅长：需求理解、决策判断、质量评估            │
│                                                      │
│   当前架构：让 LLM 做它不擅长的事                     │
│              并用它擅长的事来伪装                      │
│                                                      │
│   结果：产出物格式完美，但质量堪忧                    │
│                                                      │
└──────────────────────────────────────────────────────┘
```

具体表现：

| 能力 | LLM 实际表现 | 架构期望 |
|------|-------------|---------|
| 需求理解 | 基于关键词联想，而非真实理解 | 深度分析用户意图 |
| 决策判断 | 基于概率生成最可能的答案 | 基于上下文做出最优决策 |
| 质量评估 | 检查格式是否正确 | 评估功能是否可用 |
| 格式生成 | 非常擅长 | ✅ 匹配 |
| 模板填充 | 非常擅长 | ✅ 匹配 |
| 代码补全 | 非常擅长 | ✅ 匹配 |

**核心洞察**：当前架构把 LLM 当成了"全能工人"，但实际上 LLM 只是"高级模板引擎"。真正的需求分析、设计决策、质量评估，需要人类的判断力和领域知识。

## 五、架构改进方向（仅分析，不实施）

### 方向 1：从"流水线"到"协作网络"

```
当前:  user → PM → PRD → Design → Code → QA → done

改进:  user ←→ [PM + Product + Design] ←→ [Code + QA]
              ↕ 共享上下文              ↕ 持续集成
```

### 方向 2：从"产出物导向"到"能力导向"

| 旧定义 | 新定义 |
|--------|--------|
| product-manager: PRD 输出 | 需求分析师：验证假设、定义边界、输出用户故事 |
| ui-designer: 设计规范 | 视觉设计师：输出线框图、定义交互、验证可用性 |
| frontend-expert: 代码生成 | 工程师：迭代开发、单元测试、代码审查 |
| qa-engineer: 质量验证 | 测试工程师：编写测试、功能验证、回归检查 |

### 方向 3：从"形式化确认"到"可视化共创"

| 确认节点 | 旧方式 | 新方式 |
|---------|--------|--------|
| 计划确认 | 文字描述 | 用户故事地图 |
| PRD 确认 | 摘要表格 | 交互式原型链接 |
| 设计确认 | 摘要表格 | 线框图/高保真预览 |
| 代码确认 | 无 | 实时预览 + 交互测试 |

## 六、终极问题：这个架构到底在解决什么问题？

当前架构试图解决的是**"如何让 AI 自动完成软件开发"**，但实际上：

- 软件开发的核心不是"写代码"，而是"理解需求"和"做出决策"
- LLM 擅长的是"执行已知方案"，而非"探索未知问题"
- 当前架构把最难的"需求理解"交给了最弱的环节（product-manager 模板填充）
- 把最简单的"代码生成"交给了最强的环节（frontend-expert 批量生成）

**结果就是：代码写得很快，但可能不是用户想要的。**

---

## 七、一句话总结

> **当前架构是一个"格式正确的空壳"——每个 Agent 都完美地完成了自己的格式要求，但没有人真正关心用户想要什么。问题不在于 Agent 不够聪明，而在于架构设计把"软件交付"简化成了"文件传递"，把"质量保证"简化成了"构建通过"，把"用户确认"简化成了"回复确认"。真正的多 Agent 协作，应该是一个共享心智模型的智能网络，而非一条盲目的流水线。**
